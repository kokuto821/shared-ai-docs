# 概要

ユーザーが学習したことや、システムとの対話で得られた有用な知見、およびユーザーからの指摘・フィードバック（誤りの訂正や改善提案を含む）を整理し、ローカルのナレッジベースディレクトリにカテゴリごとにマークダウン（.md）ファイルとして保存・更新・管理します。

# ナレッジの保存先ディレクトリ

デフォルトのベースディレクトリ: `c:/Users/ihcia/.gemini/my-memo/knowledge/`
（※ユーザーから明示的に別のパスが指定された場合は、そのパスをベースディレクトリとして使用してください）

# 実行手順

1. **トピックの分析とカテゴリ設定**
   - ユーザーから提供された学習内容や知見、またはユーザーからの指摘事項やフィードバック内容を読み解きます。
   - 新規記録か既存ナレッジの更新かを判断し、以下のいずれかのうち内容に最も適した**カテゴリ（ディレクトリ）名**を選択します。
     - `coding` : 実装、プログラミング言語ごとのTips、エラー対応など（コーディング）
     - `architecture` : システムアーキテクチャ、データモデリング、設計原則など（設計）
     - `research` : 技術調査、ツール選定の比較、新技術の検証など（調査）
     - `design` : UI/UXデザイン、Figmaの使い方、スタイリングなど（デザイン）
     - `documentation` : 仕様書の書き方、ドキュメントの構成、記事作成など（文書作成）
   - 内容を端的に表す**ファイル名**を決定します。
     - _例: `git_rebase_interactive.md`, `react_hooks_memo.md` など_

2. **マークダウンコンテンツの作成**
   以下のフォーマットに沿って、構造化された分かりやすいマークダウンを作成します。

   ```markdown
   # [ナレッジのタイトル]

   **作成日**: YYYY-MM-DD
   **カテゴリ**: [カテゴリ名]
   **タグ**: [#tag1, #tag2]

   ## 概要

   [学習内容や知見の簡潔な要約・結論（3〜5行程度）]

   ## 詳細

   [以下のような要素を含めて具体的に記述する]

   - 問題の背景
   - ユーザーからの指摘事項やフィードバック（該当する場合）
   - 具体的な解決手順や改善点、気づき
   - コードスニペット（必要に応じて）
   - なぜそのアプローチや修正を取るのかの理由

   ## 参考・関連情報

   - [関連するKIs、ドキュメントのリンク、会話の履歴など（あれば）]
   ```

3. **ファイルの書き込み**
   - 決定したカテゴリ名とファイル名に基づき、ファイルの絶対パスを構築します。
     - パス例: `c:/Users/ihcia/.agents/knowledge/<カテゴリ名>/<ファイル名>.md`
   - `write_to_file` ツールを使用してファイルを保存します（ディレクトリが存在しない場合はツールが自動で作成します）。ファイルの上書きが必要な場合は、ユーザーに確認をもらうか `Overwrite: true` にして更新します。

4. **ユーザーへの結果報告**
   - 保存先のパス（カテゴリ/ファイル名）と、登録したナレッジの簡単なサマリーをユーザーに報告し、記録が完了した旨を伝えます。
