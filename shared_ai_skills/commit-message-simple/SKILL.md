---

# 目的

ユーザーから提供された変更内容や指示をもとに、一貫したフォーマットの日本語コミットメッセージを生成する。同時に、複数の異なる意味を持つ変更が含まれている場合は、それらを適切に論理的な単位で分割して提案する。

# コミット分割のルール

- 1つのコミットには1つの論理的な変更のみを含める（例：「機能追加」と「無関係なバグ修正」、または「リファクタリング」と「機能追加」などを分ける）。
- 変更点（diff）の中に複数の意図が混在している場合、プレフィックスごとにコミットを分割すること。

# フォーマット

各コミットメッセージは以下の形式に厳密に従うこと。

`<prefix>: <○○>のため<△△>する`

- `<prefix>`: 下記のプレフィックス一覧から適切なものを一つ選択する。
- `<○○>`: 変更の「目的」や「理由」を簡潔に記載する。
- `<△△>`: 具体的な変更の「内容」や「手段」を簡潔に記載する。

# プレフィックス一覧

- `feat`: 新しい機能を追加
- `fix`: バグ修正
- `refactor`: コードのリファクタリング
- `style`: コードの見た目やフォーマットを修正 (機能に影響なし)
- `test`: テストコードの追加や修正
- `docs`: ドキュメントの修正や追加
- `chore`: ビルドやツール関連の変更
- `remove`: ファイルや機能の削除
- `rename`: ファイル名の変更

# 指示

1. 対象となる変更内容（diffや説明）を分析し、変更の意図が複数混ざっていないか（分割すべきか）を判断する。
2. 分割が必要な場合は、対象ファイルや変更内容を論理的なグループに分ける。
3. グループごとに最も適したプレフィックスを選択する。
4. グループごとに、変更が「なぜ行われたのか（○○）」と「何を行ったのか（△△）」を把握する。
5. `<prefix>: <○○>のため<△△>する` という構成の1行のメッセージを作成する。
6. 分割の有無にかかわらず、**必ず実際のコミット処理を行う前**に、「どのようなファイル構成で、どのようなコミットメッセージにするか」のコミット計画を日本語でユーザーに提案する。
   - 提案フォーマットの例:
     - コミット1（対象ファイル: `A.ts`, `B.ts`）
       - メッセージ: `feat: ユーザー管理のためログイン機能を追加する`
     - コミット2（対象ファイル: `C.css`）
       - メッセージ: `style: 視認性向上のためボタンの色を変更する`
7. ユーザーから構成の提案に対する承認（OKなど）を得た後にのみ、後続の操作（`git commit` コマンドの提示や実行など）を進めること。
